= Restore workloads with clean restore in NetApp Ransomware Resilience
:hardbreaks:
:nofooter:
:icons: font
:linkattrs:
:imagesdir: ./media/

[.lead]
With a clean restore, NetApp Ransomware Resilience enables you to create an isolated recovery environment that sequesters the workload to identify and clean affected files. The clean restore identifies optimized recovery paths to help minimize data loss and get your workloads back online in the shortest amount of time possible.

The clean restore option is an alternative to the xref:custom-restore-task.html[custom restore option]. 

== Overview 

Clean recovery guides you through multi-step process that optimizes recovery. 

* **Setup**: Clean recovery requires an isolated recovery environment. When you initiate a clean recovery, you select the isolated recovery environment, then Ransomware Resilience creates a clean room where all the subsequent operations for the workload are conducted. 
* **Analysis**: Ransomware Resilience investigates the workload for malware and reviews all restore points—snapshots and backups—from the seven days prior to the incident to recommend a recovery path. 
+
Once a restore point is discovered without impacted files, all earlier restore points are skipped. For example, if the ransomware event occurred on October 10, and a restore point was discovered on October 7 at 10:21 AM with no impacted files, all restore points prior to October 7 at 10:21 AM are skipped. 
+
The analysis step also displays the number of files reviewed, denoting which are impacted and which are not. 
* **Plan**: After the analysis, Ransomware Resilience suggests two recovery plans:
+
** Least data loss
** Latest unimpacted restore point.
+
In the planning stage, you can compare the different restore plans. The planning menu captures the amount of data loss in hours for every file in the restore plan, the recommended restore point and date. You can also view file histories in the planning step, to see when ransomware events impacted the files and how the recovery point relates in time to that event. 
* **Cleaning**: Ransomware Resilience iterates through every file in the impacted environment, cleaning files infected with malware. If any files can't be cleaned, they are excluded from the recovery and quarantined to a separate location. 
* **Recovery**: Ransomware Resilience recovers the workload to its original destination. You also have the option to save the original workload before releasing the workloads and returning to normal operations. 
* **End**: Ransomware Resilience provides a detailed summary of the process and closes the clean room, freeing the isolated recovery room for other clean restores.  

== Supported configurations

You can create an isolated recovery environment in the cloud or on-premises. Currently, the following configurations are supported:

[%header]
|===
| Source | Destination (isolated recovery environment)
| On-premises | On-premises (VMware vCenter)
| On-premises | Cloud: AWS in US East 1 or EU Central 1
| Cloud: Cloud Volumes ONTAP with AWS | Cloud: AWS in US East 1 or EU Central 1
|===

* For Cloud Volumes ONTAP, the Console agent and Cloud Volumes ONTAP instance must be set up in a dedicated virtual private cloud. 
* You must be running ONTAP 9.15.1 or later. 
* On-premises sources must be AFF or FAS systems. 
* For AWS deployments, you must have administrator access to the AWS account. AWS credentials must provide full access to create the isolated recovery environment. 

// SOURCE: Amazon FSx for NetApp ONTAP, Google Cloud NetApp Volumes, Cloud Volumes ONTAP with Azure; Cloud Volumes ONTAP with Google Cloud. 

== Considerations

* Clean restore is currently only supported for NAS file shares (NFS/CIFS).
* You can only restore to the original environment in the source. 
// * Number of isolated recovery environments per console agent
* If the isolated recovery environment doesn't have capacity for a new operation, it's queued until there is availability. 
* When you configure an isolated recovery environment, it's connected to a Console agent. Workloads must be cleaned in an isolated recovery environment hosted on the same Console agent. 
* You can monitor the status of active and queued clean restore operations at any time in the **Recovery** dashboard of Ransomware Resilience.

== Create an isolated recovery environment

Before conducting a clean restore, you must create an isolated recovery room. 

The process for creating an isolated recovery room differs if the environment is in the cloud or on-premises. Ensure you follow the instructions for the correct location. 

[role="tabbed-block"]
====
.Cloud
--
. In Ransomware Resilience, select **Settings**. 
. In the Clean restores card, select **Add** if this is your first environment or **Manage** if you've already created one. 
. If this is your first time creating an isolated recovery environment, proceed to the next step. 
If you've already created an isolated recovery environment, select **Add** to create another one. 
. For a cloud-based isolated recovery environment, select **Amazon Web Services**. 
. Expand the **Prerequisites** section then select **View prerequisites**. Choose the architecture of your isolated recovery environment (for example, cloud to cloud), then review the prerequisites. 
. Expand the **Name** section. Create a **Name** for the environment. 
. Expand the *Systems* section. Choose the systems that you want to connect to the isolated recovery environment. For each system you select, you must provide the storage VM's IP address, subnet mask, and gateway.
+ 
image:clean-restore-systems.png[Screenshot of systems selection interface.]
. Expand the **Authentication** section. 
    * Enter the cloud **Account** ID and select the **Region** for the account. 
    * Enter the **Access key** and **Secret key** for the account**. 
* Expand the *Virtual private cloud (VPC)* section. Enter the VPC ID where the isolated recovery environment is to be deployed. 
+
image:screenshot-set-up-recovery-environment.png[Screenshot of isolated recovery environment authentication set-up.]
+
. Select **Add** to create the isolated recovery environment.
--
.On-premises
--
. In Ransomware Resilience, select **Settings**. 
. In the Clean restores card, select **Add** if this is your first environment or **Manage** if you've already created one. 
. If this is your first time creating an isolated recovery environment, proceed to the next step. 
If you've already created an isolated recovery environment, select **Add** to create another one. 
. In the location block, select **On-premises**. 
. Expand the **Prerequisites** section then select **View prerequisites** then review the prerequisites. 
. Expand the *Name* section. Assign a **Name** for the isolated recovery environment. 
. Expand the *Systems* section. Choose the systems that you want to connect to the isolated recovery environment. For each system you select, you must provide the storage VM's IP address, subnet mask, and gateway. 
+
image:clean-restore-systems.png[Screenshot of systems selection interface.]
. Expand the **Console agent** section. Select the Console agent from the dropdown menu that's deployed in the isolated recovery environment. 
. Expand the *Compute* section. 
.. Provide the credentials to authenticate the vCenter server: *IP address* or full-qualified domain name (FQDN), *Port*, *User name*, and *Password*.
.. Select **Authenticate** to confirm the credentials. You must receive confirmation hat the credentials are authenticated before proceeding. 
.. Select the *Datacenter*, *Cluster*, and *ESXi host* where the VMs will be deployed.
.. Select the ESXi hosts *Datastore* and *Network*.
.. Enter the *IP address range* and *Gateway* for the VMs. 
+
image:clean-restore-compute.png[Screenshot of compute selection interface.]
. Expand the *VMs* section, which is populated based on your inputs in the *Compute* section. Select the VMs located on the ESXi host that you will use for scanning malware during the clean recovery. Provide the *User name* and *Password* for each VM you select. 
. Select **Add** to create the isolated recovery environment.
--
====

You can track the progress. In the settings tab, select **Isolated recovery environments**. The isolated recovery environments page displays all isolated recovery environments and their status. Select **View jobs** for a breakdown of all jobs pertaining to that environment. 

You can modify the systems in an isolated recovery environment after it's created. In the settings tab, select **Isolated recovery environments**. Locate the isolated recovery environment you want to edit, select the action menu `...` then **Edit**. Proceed to modify the systems. Select **Save** when complete. 

== Perform a clean restore

. In the **Recovery** section, select the workload you want to restore. Select **Restore**. 
. In the Restore Type options, select **Clean restore** then **Next**. 
. Select the isolated recovery environment you want to use then **Next**. 
. Review the isolated recovery environment, confirming it's the correct location. Select **Restore**. 
// The isolated recovery environment is automatically selected if one is already connected to the Console agent. Otherwise, you can select an isolated recovery environment
. If you haven't peered the isolated recovery environment, accept the peering request in the production environment.
. After the peering request is accepted, Ransomware Resilience performs the necessary set up. Once the set up completes successfully, Ransomware Resilience denotes **Setup complete**. To proceed to data analysis, select **Next**. 
. After the setup completes, select **Run analysis**. Ransomware Resilience analyzes all available restore points in the seven days prior to the ransomware event, displaying the date and type of restore point. 
+
image:screenshot-clean-room-analysis.png[Screenshot of the clean restore analysis step.]

. After the analysis completes, select **Next** to plan your recovery. Ransomware Resilience presents two options: **Least data loss** and **Latest unimpacted restore point**. Select either option, and optionally look at individual file data for either restore point. 
+
To view granular file event data, select a file to analyze its encryption status; when it was created, modified, or deleted, and what restore point corresponds to what action. 
. After you choose your restore point, select **Create restore point** to begin cleaning your files. 
. Ransomware Resilience begins cleaning the workload. 
//If cleaning of any file fails, those files are excluded from the recovery. Select if you want to quarantine those files by saving them to another location. 
+
When the cleaning completes, select **Next** to initiate the recovery. 
. Choose where to save the original workload: the original location or an alternate. 
+
If you choose the original location, select whether you want to save a new copy or replace the existing one. 
+
If you select an alternative location, select the **System** and **Storage VM** then enter the **Path for the new volume** and the **New volume name**. 
. Select **Start recovery** to initiate the recovery. 
. Once the recovery completes, you can release the resource based on the save location from the previous step. Select **Release resources and end** to initiate this process. To confirm you want to release the resource, select **End**. 

== Delete an isolated recovery environment

You can't delete a clean room with an restoration in progress. You can cancel the restore operation or wait until the restore completes then delete the clean room. 

. Go to **Settings**.
. In the Clean restore card, select **Manage**. 
. Identify the clean room you want to delete. In the actions menu column for that room, select the three dots `...` then **Delete**.
. In the dialog, select **Delete** to confirm the operation.
