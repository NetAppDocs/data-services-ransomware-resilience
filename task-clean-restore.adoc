= Configure the clean restore environment in NetApp Ransomware Resilience
:hardbreaks:
:nofooter:
:icons: font
:linkattrs:
:imagesdir: ./media/

[.lead]
NetApp Ransomware Resilience offers clean restores, which provide a guided recovery after ransomware attacks. To perform a clean restore, you must first create an isolated recovery environment (IRE), either on-premises or in a supported cloud environment. In the isolated recovery environment, you create a clean room where Ransomware Resilience sequesters the workload to identify which files are clean and which are impacted by ransomware.

The clean restore option is an alternative to the xref:task-custom-restore.html[custom restore option]. 

include::_include/console-role-admin.adoc[]

== Overview 

Clean restore guides you through multi-step process that optimizes recovery. 

* **SETUP**: To perform a clean restore, you must first create an isolated recovery environment then select the workload to restore. Ransomware Resilience creates a clean room where all actions for the workload are taken. 
* **ANALYSIS**: In the clean room, Ransomware Resilience analyzes all snapshots to see if encryption is present. Clean restore determines a recovery map to determine the optimal recovery plan. 
+
Once a restore point is discovered without impacted files, all earlier restore points are skipped. For example, if the ransomware event occurred on October 10, and a restore point was discovered on October 7 at 10:21 AM with no impacted files, all restore points prior to October 7 at 10:21 AM are skipped. 
+
The analysis step also displays the number of files reviewed, denoting which are impacted and which are not. 
* **PLAN**: Choose the provided options for recovery:
** _Latest unimpacted restore point_: the quickest restore point from the most recent unecrypted snapshot from before the attack
** _Least data loss_: the most recent version of every unecrypted file from different snapshots
+
You can also view file histories in the planning step, to see when ransomware events impacted the files and how the recovery point relates in time to that event. 
* **CLEANING**: Ransomware Resilience removes malware from the recovery point. If any files can't be cleaned, they are excluded from the recovery and quarantined to a separate location. 
* **RESTORE**: Ransomware Resilience restores the clean data into the source environment. 
* **END**: Ransomware Resilience provides a detailed summary of the process and closes the clean room.

== Considerations

include::_include/clean-restore-considerations.adoc[]

=== On-premises considerations 

If you're deploying an on-premises IRE, when a Windows VM is cloned for the recovery analysis in the clean restore, the cloned VM retains the same configuration as the source. This can cause conflicts:

* Using the same security identfier (SID) results in authentication and security conflicts.
* Using the same computer name (CN) causes network conflicts. 
* Using the same machine identity leads to licensing and activation issues. 

To avoid these conflicts, Ransomware Resilience performs sysprem on the clone VM, which resets the SID, CN, and machine identity. Resetting these values ensures the clone VM operates as a unique and independent instance without interfering with the source VM. 

=== Secondary sources 

When you create the isolated environment for on-premises sources, you have the option of adding a secondary source. 

If the primary source is available, the clean restore must be restored to the primacy source. If the primary source is unavailable and you've configured a secondary source, clean restore will restore to the secondary source. 

The clean restore process by default analyzes snapshots from the source environment. If either the source environment is unavailable or an unencrypted snapshot on the source is not available, clean restore analyzes snapshots on a secondary system _if you have configured one_. 

== Cost and licensing

Clean restore is offered as part of Ransomware Resilience. No additional license is required to perform a clean restore or create an isolated recovery enviornment. 

Enabling clean restore can incur third-party cloud provider charges. Depending on how you use the service, these charges can recur while the clean recovery environments exist.

Third-party charges can include creation and deployment of compute instances and additional production storage capacity for cleaning and recovery. For example, when IRE is deployed in AWS and the clean restore is initiated, two t3.medium AWS EC2 instances are deployed in the clean room within the isolated recovery environment for cleaning encryption and malware. 

Clean recovery in Cloud Volumes ONTAP instances creates a clean room storage VM for metadata volumes and snapshot clones for analysis.

== Create an isolated recovery environment

Before conducting a clean restore, you must create an isolated recovery room. 

The process for creating an isolated recovery room differs if the environment is in the cloud or on-premises. Ensure you follow the instructions for the correct location. 

[role="tabbed-block"]
====
.Cloud
--
. In Ransomware Resilience, select **Settings**. 
. In the Clean restores card, select **Configure** if this is your first environment or **Manage** if you've already created one. 
. If this is your first time creating an isolated recovery environment, proceed to the next step. 
+
If you've already created an isolated recovery environment, select **Add** to create another one. 
// pre-req step. Choose on-premises then review and confirm. 
. For a cloud-based isolated recovery environment, select **Amazon Web Services**. 
. Expand the **Name** section. Create a **Name** for the environment. 
. Expand the *Systems* section. Choose the systems you want to connect to the isolated recovery environment. For each system you select, you must provide the storage VM's IP address, subnet mask, and gateway.
+
[NOTE]
For isolated recovery environments deployed to Amazon FSxN for ONTAP, you don't need to provide this information. 
+ 
image:clean-restore-systems.png[Screenshot of systems selection interface.]
. Expand the **Authentication** section. 
    * Enter the cloud **Account** ID and select the **Region** for the account. 
    * Enter the **Access key** and **Secret key** for the account**. 
* Expand the *Virtual private cloud (VPC)* section. Enter the VPC ID where the isolated recovery environment is to be deployed. 
+
image:screenshot-set-up-recovery-environment.png[Screenshot of isolated recovery environment authentication set-up.]
+
. Select **Add** to create the isolated recovery environment. Ransomware Resilience tests the connection after you select add; This process can take a while.

== Delete an isolated recovery environment

You can't delete a clean room with an restoration in progress. You can cancel the restore operation or wait until the restore completes then delete the clean room. 

[IMPORTANT]
If you delete the IRE, it also delete the clean room SVM and the metadata volume. Once these assets have been deleted, you are no longer able to generate reports for the clean restore. 

. Go to **Settings**.
. In the Clean restore card, select **Manage**. 
. Identify the clean room you want to delete. In the actions menu column for that room, select the three dots `...` then **Delete**.
. In the dialog, select **Delete** to confirm the operation.
